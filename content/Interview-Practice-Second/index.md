## 호이스팅에 대해 설명해보세요

호이스팅(Hoisting)은 JavaScript에서 변수와 함수 선언이 해당 스코프의 최상단으로 끌어올려지는 것처럼 동작하는 특성을 의미합니다. 이는 변수와 함수가 실제 코드에서 선언된 위치와는 상관없이, 실행 전에 먼저 처리되기 때문입니다. 이로 인해 변수를 선언하기 전에 참조할 수 있는 것처럼 보입니다.

## "==" 와 "==="의 차이는 무엇인가요?

### 동등 연산자 "=="

- 형변환을 수행: `==`는 두 값을 비교할 때, 필요하면 서로 다른 데이터 타입을 동일한 타입으로 변환하여 비교합니다. 이 과정을 "형변환" 또는 "타입 강제 변환(type coercion)"이라고 합니다.
- 예측 불가능한 결과 가능: 형변환이 자동으로 이루어지기 때문에, 때로는 예측하지 못한 결과를 초래할 수 있습니다.

### 일치 연산자 "==="

- 형변환을 수행하지 않음: `===`는 두 값을 비교할 때 형변환을 수행하지 않습니다. 값과 데이터 타입이 모두 같은 경우에만 참을 반환합니다.
- 예측 가능한 결과: 형변환 없이 비교하기 때문에, 항상 예측 가능한 결과를 제공합니다.

## 클로저란 무엇인가요?

> 클로저(Closure)는 함수가 선언될 때 생성되는 렉시컬 환경(Lexical Environment)을 참조하여, 함수가 선언될 당시의 스코프를 기억하고, 외부 함수가 실행되고 반환된 후에도 이 스코프에 접근할 수 있는 함수입니다. 클로저는 JavaScript에서 중요한 개념으로, 정보를 은닉하고 특정 데이터에 대한 접근을 제어하는 데 주로 사용됩니다.

### 렉시컬 환경과 스코프 체인

함수가 선언될 때마다 함수의 렉시컬 환경이 생성됩니다. 이 렉시컬 환경은 함수가 선언된 위치에서의 변수와 상수를 포함합니다. 클로저는 이 렉시컬 환경을 기억하여, 함수가 선언된 위치의 스코프를 참조할 수 있습니다.

### 클로저의 특징

1. 상태 유지: 클로저는 함수가 선언될 당시의 변수 값을 기억하고 유지합니다.
2. 은닉화: 클로저를 사용하면 특정 변수나 함수를 외부에서 접근하지 못하게 하여, 데이터 은닉과 캡슐화를 구현할 수 있습니다.
3. 스코프 체인: 클로저는 함수가 선언될 때의 스코프 체인을 따라 외부 함수의 변수에 접근할 수 있습니다.

## 클래스형 컴포넌트와 함수형 컴포넌트의 차이는 무엇일까요?

1. 상태 및 생명주기 다루는 방식:
   - 클래스형 컴포넌트: 상태를 `this.state`로 정의하고, 생명주기 메서드를 오버라이드하여 다양한 작업을 수행합니다. 예를 들어, `componentDidMount`, `componentDidUpdate`, `componentWillUnmount` 등의 생명주기 메서드를 사용하여 컴포넌트의 초기화, 업데이트, 제거 등의 작업을 수행합니다.
   - 함수형 컴포넌트: 상태를 `useState` 훅을 사용하여 정의하고, 생명주기를 `useEffect` 훅을 사용하여 다룹니다. `useEffect` 훅은 `componentDidMount`, `componentDidUpdate`, `componentWillUnmount` 등의 생명주기 메서드 역할을 수행할 수 있습니다.
2. 문법적인 부분:
   - 클래스형 컴포넌트: ES6의 클래스(class)를 사용하여 정의됩니다.
   - 함수형 컴포넌트: ES6의 화살표 함수(arrow function)를 사용하여 정의됩니다.
3. 참조 방식:
   - 클래스형 컴포넌트: 상태나 메서드를 참조할 때 `this` 키워드를 사용합니다.
   - 함수형 컴포넌트: 상태나 메서드를 참조할 때 `this` 키워드를 사용하지 않습니다.
4. 장점:
   - 클래스형 컴포넌트: 복잡한 상태 관리나 생명주기에 대한 세밀한 제어가 필요할 때 유용합니다.
   - 함수형 컴포넌트: 간결하고 가독성이 좋으며, 테스트와 리팩토링이 쉽습니다. 또한, 훅 API를 사용하여 상태 및 생명주기를 다룰 수 있어서 클래스형 컴포넌트에 비해 코드가 간결해집니다.

## 이벤트 버블링에 대해서 설명해주세요.

- **이벤트 버블링**은 HTML에서 이벤트가 발생한 요소에서부터 상위 요소로 이벤트가 전파되는 현상을 말합니다. 이는 이벤트가 발생한 요소에서 시작하여 부모 요소, 그 부모의 부모 요소로 이벤트가 계속해서 전달되는 것을 의미합니다. 예를 들어, 버튼을 클릭할 때 클릭 이벤트가 발생하면 해당 버튼의 부모 요소, 그 부모의 부모 요소로 이벤트가 전파됩니다. 부모 요소에 등록된 이벤트 핸들러가 있다면 해당 요소에서도 이벤트를 감지할 수 있습니다. 이를 활용하여 여러 개의 하위 요소에서 발생하는 이벤트를 하나의 이벤트 핸들러로 처리하는 **이벤트 위임**이 가능합니다.
- **이벤트 캡처링**은 이벤트 버블링과 반대로 상위 요소에서 하위 요소로 이벤트가 전파되는 현상을 말합니다. 이벤트 캡처링은 이벤트가 발생한 요소의 최상위 조상부터 이벤트가 발생한 요소까지 이벤트가 전파됩니다. 그 후에 이벤트 버블링이 발생합니다.
- **이벤트 위임**은 이벤트 버블링을 활용하여 여러 개의 하위 요소에서 발생하는 이벤트를 하나의 상위 요소에 등록된 이벤트 핸들러로 처리하는 기법입니다. 이를 통해 동적으로 생성된 요소에도 이벤트를 적용할 수 있고, 메모리 사용을 줄일 수 있습니다. 이벤트 위임은 성능 향상과 코드 간결성을 높이는 데 유용합니다.
- **이벤트 버블링**과 **이벤트 캡처링**의 주요 차이점은 이벤트 전파의 방향입니다. 이벤트 버블링은 하위 요소에서 상위 요소로 이벤트가 전파되는 반면, 이벤트 캡처링은 상위 요소에서 하위 요소로 이벤트가 전파됩니다. 또한, 이벤트 캡처링은 이벤트 버블링 전에 발생하므로 이벤트가 발생하는 요소의 상위 요소에서 이벤트를 감지할 수 있습니다.
- 따라서, 이벤트 버블링과 이벤트 캡처링은 서로 보완적인 개념으로 이해되어야 합니다. 이벤트 버블링은 주로 하위 요소에서 상위 요소로 이벤트를 전파할 때 사용되고, 이벤트 캡처링은 이벤트가 발생한 요소의 상위 요소에서 하위 요소로 이벤트를 전파할 때 사용됩니다.
