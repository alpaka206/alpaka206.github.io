## React를 선택한 이유가 있을까요?

저는 React를 사용하는 것을 선호합니다. React의 핵심 요소인 Virtual DOM이라고 생각합니다. 웹 브라우저가 텍스트 문서를 읽어서 DOM구조를 바꾸어 사용자에게 보내주는데 반응에 따라 전체화면을 새로 그리는 것은 성능에 좋지 않기때문에 불필요한 화면의 갱신을 최소화하여 성능을 향상시키는 Virtual Dom이 React를 선택한 이유입니다.<br/>
또한 완성도가 높고 이해하기 쉬운 개발 워크플로우입니다. React는 라이브러리에 최적화된 개발 인터페이스와 코딩 언어입니다. Vue와 같은 프레임워크와 달리 불필요한 HTML 어트리뷰트가 없기에 간결하고 가독성이 뛰어나다는 강점이 있다고 생각합니다

## RESTful API에 대해 설명해주세요

RESTful API는 HTTP 프로토콜을 기반으로 하는 웹 서비스 아키텍처입니다. REST(Representational State Transfer)는 클라이언트와 서버 간의 통신을 위한 자원, 메소드, 메시지 등을 정의합니다. RESTful API는 표준 HTTP 메소드(GET, POST, PUT, DELETE)를 사용하여 서버와 통신하며, 이를 통해 자원을 생성, 조회, 업데이트, 삭제할 수 있습니다.

## HTTP 메소드에 대한 설명해주세요

- **GET**: 서버에서 리소스(데이터)를 요청하는 메소드입니다. 데이터를 조회하여 응답합니다.
- **POST**: 서버에 데이터를 전송하는 메소드입니다. 새로운 데이터를 생성하여 서버에 저장하고, 결과를 응답합니다.
- **PUT**: 서버의 기존 데이터를 업데이트하는 메소드입니다. 지정된 데이터를 수정하여 저장하고, 결과를 응답합니다.
- **DELETE**: 서버에서 데이터를 삭제하는 메소드입니다. 지정된 데이터를 서버에서 제거하고, 결과를 응답합니다

## CSRF나 XSS 공격을 막는 방법은?

### CSRF 공격 방지 방법

> 사용자가 의도하지 않은 요청을 보내도록 속여 웹 애플리케이션에서 권한을 가진 사용자의 행동을 강제로 수행하게 만드는 공격

- **CSRF 토큰 사용**: 서버에서 CSRF 토큰을 생성하여 세션에 저장하고, 프론트엔드에서 요청 시 이 토큰을 함께 전송하여 인증합니다. 이를 통해 요청이 신뢰할 수 있는 출처에서 온 것인지 확인할 수 있습니다.
- **SameSite 속성 설정**: 쿠키에 SameSite 속성을 설정하여, 다른 도메인에서는 쿠키를 사용할 수 없도록 제한합니다. 이를 통해 다른 사이트에서 악의적인 요청을 보내는 것을 방지합니다.

### XSS 공격 방지 방법

> 웹 애플리케이션의 취약점을 이용하여 악성 스크립트를 주입하는 공격, 사용자가 입력한 데이터를 제대로 검증하지 않거나, 출력 시 인코딩하지 않을 때 발생

- **입력 값 유효성 검증**: 사용자로부터 입력받은 값들을 철저히 검증하고, 특수 문자를 제거하거나 인코딩합니다.
- **정규식 사용**: 입력값에서 특수 문자를 제외하는 정규식을 사용하여 악성 스크립트가 포함되지 않도록 합니다.
- **CSP(Content Security Policy) 설정**: 서버에서 CSP를 설정하여, 허용된 스크립트만 실행되도록 제한합니다. 이를 통해 임의의 스크립트 실행을 방지할 수 있습니다.
- **HTTPS 사용**: HTTP 대신 HTTPS를 사용하여 통신을 암호화함으로써, 중간자 공격을 방지하고 데이터의 무결성을 보장합니다.

## async/await에 대해 설명해보세요.

async/await는 비동기적인 작업을 처리할 수 있는 문법입니다. 비동기 함수는 **`async`** 키워드로 정의되며, 함수 내부에서 **`await`** 키워드를 사용하여 비동기 작업이 완료될 때까지 기다린 후 결과값을 반환할 수 있습니다. **`async/await`**는 Promise를 기반으로 하며, 코드를 더 간결하고 직관적으로 작성할 수 있도록 해줍니다.

## 브라우저 렌더링 원리에 대해서 설명해보세요.

> 브라우저 렌더링은 HTML, CSS, JavaScript 등의 웹 페이지 자원을 브라우저가 화면에 그리는 과정을 말합니다. 브라우저 렌더링 원리와 순서는 크게 다음과 같은 단계로 구성됩니다:

### 1. DOM 생성

- 브라우저는 HTML 문서를 파싱하여 DOM(Document Object Model) 트리를 생성합니다. 이 과정에서 HTML 태그를 노드로 변환하고, 노드 간의 계층 관계를 형성합니다. 각 HTML 태그는 DOM 트리의 노드로 변환되며, 이 트리는 페이지 구조를 나타냅니다.

### 2. CSSOM 생성

- 브라우저는 CSS 파일을 파싱하여 CSSOM(CSS Object Model) 트리를 생성합니다. 이 과정에서 CSS 속성을 노드로 변환하고, 노드 간의 계층 관계를 형성합니다. CSSOM 트리는 스타일 규칙을 나타내며, DOM 트리와 결합하여 요소의 스타일을 결정합니다.

### 3. 렌더 트리 생성

- 브라우저는 DOM 트리와 CSSOM 트리를 결합하여 렌더 트리(Render Tree)를 생성합니다. 렌더 트리는 실제 화면에 표시될 요소만을 포함하며, 각 요소가 어떻게 스타일링되고 배치되는지를 나타냅니다. 이 과정에서 **`display: none`**과 같은 속성을 가진 요소는 렌더 트리에 포함되지 않습니다.

### 4. 레이아웃(리플로우)

- 렌더 트리가 생성되면, 브라우저는 각 요소의 크기와 위치를 계산하는 레이아웃(리플로우) 과정을 거칩니다. 이 단계에서 요소들의 너비, 높이, 위치가 계산됩니다. 레이아웃 과정은 페이지의 흐름을 결정짓고, 각 요소가 화면의 어느 위치에 그려질지를 결정합니다.

### 5. 페인팅

- 레이아웃이 완료된 후, 브라우저는 페인팅(Painting) 과정을 통해 요소를 실제 화면에 그립니다. 이 단계에서 각 요소의 배경, 테두리, 글자 등을 그리게 됩니다. 페인팅은 여러 레이어로 나뉘어 처리될 수 있으며, 최종적으로 모든 레이어가 합쳐져 사용자에게 보여집니다.

### 6. 컴포지팅

- 페인팅된 레이어들을 합성하여 최종적으로 화면에 그리는 과정입니다. 여러 레이어를 결합하여 최종적으로 사용자가 보는 화면을 완성합니다.

1. **DOM 생성**: HTML 파싱 -> DOM 트리 생성
2. **CSSOM 생성**: CSS 파싱 -> CSSOM 트리 생성
3. **렌더 트리 생성**: DOM + CSSOM -> 렌더 트리 생성
4. **레이아웃(리플로우)**: 렌더 트리 -> 요소의 크기와 위치 계산
5. **페인팅**: 요소를 화면에 그림
6. **컴포지팅**: 여러 레이어를 결합하여 최종 화면 구성
